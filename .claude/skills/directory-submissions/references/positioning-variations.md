# Positioning variations by directory type

Don't copy-paste the same description everywhere. AI engines penalize duplicate content, and each directory audience responds differently.

## Startup directories (BetaList, Product Hunt)

Lead with the **outcome**. Audience is other founders.

> "[Product] helps [audience] [verb] [outcome] without [pain]. Free to start."

Example: "Linear helps engineering teams ship faster without sacrificing quality. Free for small teams."

## SaaS / B2B review directories (G2, Capterra, SaaSHub)

Lead with the **alternative framing**. People search "[competitor] alternative".

> "[Product] is a [category] tool. Best alternative to [competitor] for [audience] who want [differentiator]."

Example: "Linear is a project management tool. Best alternative to Jira for product teams that want speed."

## AI directories (TAAFT, Futurepedia)

Lead with the **AI-first architecture**. Audience explicitly wants AI tools.

> "AI [tool] for [task]. Built on [model/approach]. [Differentiator]."

Example: "AI workspace generator for Claude Code. Built on real workflow templates. Free."

## Agent / MCP registries

Lead with the **agent or MCP angle**. Niche but high-intent.

> "[Tool] is a [agent/MCP] for [task]. Works with [Claude/Cursor/etc.]."

## No-code directories

Lead with **ease + power**. Speed-to-build over depth.

> "[Tool] lets [audience] build [output] without [coding/dev work]. [Time-to-value]."

Example: "Build complete Claude Code workspaces in 5 minutes — no terminal experience needed."

## Dev directories (Hacker News, GitHub awesome lists)

Lead with **technical depth**. Audience rewards substance.

> "[Tool] does [technical thing] using [approach]. [Open-source/specific stack]."

## B2B review sites (G2, Capterra)

Lead with **ROI + use case**. Buyers want outcomes and case studies.

> "[Product] [verb] for [team type]. [Customer X] uses it to [outcome]. [Pricing model]."

## Universal rules

- One concrete claim per variant. Avoid stuffing.
- No unverified superlatives ("best", "world-class").
- Keep all variants the same length so they're consistent without being identical.
- Always update with the live product — don't ship a variant referencing a feature you haven't built.
